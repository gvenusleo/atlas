# 协作约定

## 沟通要求

- 默认使用中文进行沟通、文档编写、代码说明与提交说明。
- 需求存在歧义时先提问，尤其是涉及数据模型、权限、安全、删除操作、关键逻辑或关键交互规则的改动。

## 技术边界

- 统一使用 Bun 作为默认工具链，安装依赖、运行脚本、调用相关 CLI 时优先使用 `bun` 或 `bunx`。
- 项目按 Next.js 全栈应用进行设计与实现，默认沿用 `app/` 目录体系与服务端优先思路。
- 后端数据持久化基于 PostgreSQL，设计表结构、查询与迁移时要优先考虑一致性、索引、可扩展性与回退成本。

## UI 与组件约束

- 所有界面组件必须使用原生 shadcn 组件。
- 如需新增组件，必须通过 Bun 命令添加，例如 `bunx --bun shadcn@latest add button`。
- 禁止手动新增、复制或改写 `components/ui` 下的 shadcn 原生组件文件。
- 禁止直接修改原生 shadcn 组件样式。
- 视觉调整应优先通过页面结构、信息层级、布局关系、文案与业务层组合完成；若必须突破该约束，先确认再实施。

## 工程要求

- 所有代码变更都必须优先考虑可测试、可维护、可扩展、高性能与长期演进成本。
- 保持模块边界清晰，避免把数据访问、视图逻辑与状态管理混杂在同一处。
- 优先编写小而清晰、职责单一、类型明确的函数与组件。
- 仅在确实需要浏览器交互时使用客户端组件，数据获取与敏感逻辑优先放在服务端。
- 避免不必要的客户端状态、重复请求、过度渲染与体积膨胀。

## 变更验证

- 任何代码改动都应补充或更新必要的验证手段，至少覆盖相关静态检查、类型检查、构建验证与必要的手工回归。
- 引入依赖、调整数据结构、修改缓存策略或异步流程时，需要明确说明动机、影响范围与回退思路。

## UI/UX 目标

- 界面风格保持简洁、干净、一致。
- 优先保证信息结构清晰、交互直观、反馈明确，而不是追求装饰性效果。
- 在可读性、可访问性与响应性能之间保持平衡。
